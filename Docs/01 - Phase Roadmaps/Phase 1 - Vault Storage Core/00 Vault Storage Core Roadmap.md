## Purpose

Build the storage foundation that allows Studium to read from and write to an Obsidian-compatible Markdown vault.

This phase answers:

> How does Studium store knowledge in a durable, portable, human-readable way?

The system should initially operate against a test vault, not the real Obsidian vault. Once the write behavior is stable, it can be connected to the real vault.

## Why This Phase Comes First

Every later feature depends on reliable vault storage.

Create, Agent Review, Backlog, Retention, and the Mastery Dashboard all require Studium to understand where notes live, how notes are structured, how metadata is stored, and how new notes are safely created.

Before Studium can intelligently reason about concepts, it needs a dependable way to read and write Markdown notes.

## Note Type Direction

Studium should support a flexible note hierarchy.

Every note should share universal fields, but different note types and subtypes may require different metadata and scaffold sections.

Initial top-level note types should include:

- concept
- reflection
- example
- unknown

Backlog should not be treated as a top-level note type.

A backlog item represents a reason that a concept needs to be learned, reviewed, or revisited. The concept itself should still be represented as a concept note or future concept note.

For example, if Chain Rule is added to the backlog because it is needed for Backpropagation, Chain Rule is still a concept. The backlog entry explains why Chain Rule is waiting to be learned.

---

## Concept Subtypes

Concept notes may eventually include subtypes such as:

- mathematical_concept
- ml_concept
- algorithm
- system_design_concept
- programming_concept
- theory_concept
- example_driven_concept

The agent should eventually help decide the correct type and subtype.

Schemas should remain flexible. Studium should warn about missing structure rather than aggressively rejecting notes. The scaffolds generated by the create agent should adhere to the schema's provided based on type. 

---

## Source Model

Each concept should support source metadata.

A concept may have a `primary_source`, which represents the main source being used to learn the concept during its initial creation.

A concept may also have `learning_encounters`, which represent later encounters with the same concept from books, classes, work, videos, papers, projects, or other contexts.

The primary source should also be represented as the learning encounter, regardless of what the source is. This gives the user an understanding of where they first learned the concept.

Example source types may include:

- book
- class
- work
- video
- paper
- project
- chatbot
- backlog
- self_study
- none

Backlog is therefore a possible **source/context type**, not a note type.

## Example Universal YAML Direction
```
yaml
id: concept_stochastic_gradient_descent_a1b2c3
note_type: concept
note_subtype: ml_concept
canonical_title: Stochastic Gradient Descent

  - SGD

status: scaffolded

  type: book
  name: Hands-On Machine Learning
  location: Chapter 4

learning_encounters:
  - type: book
    name: Hands-On Machine Learning
    location: Chapter 4
    role: primary

relationships: []

updated_at:
```

## Relationship Storage Direction 

Relationships should be stored in both machine-readable and human-readable forms. 

Machine-readable YAML is useful for parsing:
```
relationships:
  - type: prerequisite_for
    target: concept_backpropagation_x9y8z7
  - type: depends_on
    target: concept_partial_derivatives_m3n4p5
```
Human-readable Markdown is useful for the user:
```
## Relationships

- prerequisite_for: [[Backpropagation]]
- depends_on: [[Partial Derivatives]]
```

YAML should be treated as the source of truth for the system, while the Markdown section should remain readable and editable.

## Success Criteria

Phase 1 is successful when Studium can reliably create and parse Studium-compatible Markdown notes in a test vault without depending on the full Create interface.

A successful test should look like:
```
Input:
Create concept note for Stochastic Gradient Descent.

Output:
A Markdown file is created in the test vault with:
- stable concept ID
- canonical title
- note type
- note subtype
- primary source placeholder
- learning encounters section
- relationships section
- scaffold placeholder sections
```

Another successful test should look like: 
```
Input:
Create backlog-origin concept note for Chain Rule.

Output:
A Markdown file is created with:
- stable concept ID
- canonical title
- note_type: concept
- note subtype
- primary_source.type: backlog
- relationship showing that Chain Rule is needed for another concept
```

Exact fields will change according to needs, separation between Markdown and YAML, and other contributing factors as prerequisites for future phases.
